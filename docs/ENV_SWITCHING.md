# Environment Configuration Guide

**Platform:** Greene Platform (Spring Boot · Kotlin · Gradle multi-module)  
**Last updated:** April 2026

---

## Overview

The application ships with four Spring Boot configuration files:

| File | Purpose | Committed? |
|---|---|---|
| `app/src/main/resources/application.yml` | Shared base config. All env-var placeholders. | ✅ Yes |
| `app/src/main/resources/application-dev.yml` | Local developer settings. Debug logging, Swagger enabled. | ✅ Yes |
| `app/src/main/resources/application-staging.yml` | Staging server settings. Swagger enabled, try-it-out off. | ✅ Yes |
| `app/src/main/resources/application-prod.yml` | Production settings. Swagger disabled, WARN logging. | ✅ Yes |
| `.env` | Your local secrets (copied from `.env.example`). | ❌ **Never** |
| `.env.example` | Safe template — fake values only. | ✅ Yes |

> **Rule:** No credentials, passwords, keys, or tokens may ever appear in a committed file.  
> Any file matching `.env`, `.env.*`, `application-*.local.yml`, `secrets/`, `*.secret`, `*.pem`, `*.p12`, or `*.jks` is blocked by `.gitignore`.

## 1. Local Development (profile: `dev`)

### First-time setup

```bash
# 1. Copy the template
cp .env.example .env

# 2. Edit .env — fill in DB_USERNAME, DB_PASSWORD, etc.
#    For a local Docker Postgres you can keep the defaults as-is.

# 3. Start PostgreSQL and Redis (Docker Compose — see E2-US2)
docker-compose up -d postgres redis
```

### How `.env` is loaded

Spring Boot 2.4+ can natively import any `key=value` file as a property source via `spring.config.import`.  
`application-dev.yml` contains:

```yaml
spring:
  config:
    import: "optional:file:.env[.properties]"
```

The `[.properties]` hint tells Spring Boot to parse `.env` using its standard properties loader (same `KEY=VALUE` / `# comment` syntax). The `optional:` prefix means startup does not fail if `.env` is absent — useful in CI where real environment variables are set directly instead.

No extra library is needed. The file is read **before any bean is created**, so every `${...}` placeholder in the YAML files is resolved correctly.

### Running the application

**IntelliJ IDEA**

1. Open **Run / Debug Configurations** for the `:app` Spring Boot run config.
2. Set **Working directory** to the **project root** (the folder that contains `gradlew`):
   ```
   $PROJECT_DIR$
   ```
   *(IntelliJ defaults to the module directory — changing this to the root is the only required step.)*
3. No environment variables need to be set manually. `spring.profiles.default: dev` is configured in `application.yml` so the `dev` profile activates automatically when nothing else is set.

**Gradle on the command line**

```powershell
# From the project root — working directory is set to rootProject.projectDir automatically
./gradlew :app:bootRun
```

`build.gradle.kts` already sets `bootRun { workingDir = rootProject.projectDir }`, so `.env` in the repo root is always found.

## 2. Staging (profile: `staging`)

Staging is deployed automatically on every merge to `main` (GitHub Actions — see E2-US3).

### How env vars are injected

All variables are stored as **GitHub Actions secrets** (Settings → Secrets and variables → Actions) and injected at deploy time via the CI workflow:

```yaml
# .github/workflows/deploy-staging.yml (excerpt)
- name: Deploy to staging
  env:
    SPRING_PROFILES_ACTIVE: staging
    DB_URL: ${{ secrets.STAGING_DB_URL }}
    DB_USERNAME: ${{ secrets.STAGING_DB_USERNAME }}
    DB_PASSWORD: ${{ secrets.STAGING_DB_PASSWORD }}
    REDIS_HOST: ${{ secrets.STAGING_REDIS_HOST }}
    JWT_PRIVATE_KEY: ${{ secrets.STAGING_JWT_PRIVATE_KEY }}
    JWT_PUBLIC_KEY: ${{ secrets.STAGING_JWT_PUBLIC_KEY }}
    # ... remaining secrets
```

On the VPS itself the same variables are provided through the Docker Compose `.env` file that is **generated by the deployment script and never stored in the repo**.

**What the `staging` profile changes from the base config**

| Setting | Value |
|---|---|
| `spring.jpa.show-sql` | `false` |
| `logging.level.root` | `INFO` |
| Swagger UI | Enabled at `/swagger-ui` (read-only — try-it-out off) |
| DB / Redis credentials | Mandatory — application refuses to start if unset |

## 3. Production (profile: `prod`)

Production deployment is triggered manually via a GitHub Actions workflow dispatch with a confirmation step.

### How env vars are injected

Same pattern as staging but using a separate set of GitHub Actions secrets (e.g. `PROD_DB_URL`, `PROD_DB_PASSWORD`, etc.).

**What the `prod` profile changes from the base config**

| Setting | Value |
|---|---|
| `spring.jpa.show-sql` | `false` |
| `logging.level.root` | `WARN` |
| `logging.level.com.greene` | `INFO` |
| Swagger UI | **Disabled entirely** |
| OpenAPI docs (`/api-docs`) | **Disabled entirely** |
| DB / Redis credentials | Mandatory — application refuses to start if unset |

## 4. Profile Activation — Quick Reference

| Method | Command / Setting |
|---|---|
| Environment variable | `SPRING_PROFILES_ACTIVE=prod` |
| JVM system property | `-Dspring.profiles.active=prod` |
| Gradle `bootRun` arg | `--args='--spring.profiles.active=prod'` |
| `application.yml` (not recommended for secrets) | `spring.profiles.active: prod` |

> The environment variable approach (`SPRING_PROFILES_ACTIVE`) is preferred in all deployed environments because it keeps configuration outside of the codebase entirely.

## 5. Adding a New Environment Variable

1. Add the variable to `.env.example` with a **fake placeholder value** and an inline comment explaining what it is.
2. Add `${MY_NEW_VAR}` (or `${MY_NEW_VAR:safe-default}`) to the relevant profile YAML(s).
3. Commit both changes.
4. Add the real value as a GitHub Actions secret for staging and production.
5. Update your local `.env` file (not committed).

> **Never** add a real secret as a default value inside any YAML file.

## 6. Verifying No Secrets Are Committed

Run this from the repo root to scan the commit history for potential secrets:

```bash
# Using git-secrets (install first: https://github.com/awslabs/git-secrets)
git secrets --scan-history

# Quick manual grep for common patterns
git log --all --full-history -p | Select-String -Pattern "password\s*[:=]\s*\S+" -CaseSensitive
```

The pre-launch security checklist in `ARCHITECTURE.md § 7.4` also requires a full git-history secrets scan before go-live.

## 7. Troubleshooting

### Application fails to start with "Could not resolve placeholder"

A required environment variable is missing. Check which variable is named in the error, then:
- **Local:** add it to your `.env` file and restart.
- **Staging / Prod:** add it as a GitHub Actions secret and re-run the deployment workflow.

### Wrong profile is active

Check the application startup log — Spring Boot prints the active profile on startup:

```
The following 1 profile is active: "dev"
```

If the wrong profile is active, verify the `SPRING_PROFILES_ACTIVE` environment variable in your run configuration or deployment script.

### Flyway migration fails on first run

For a **brand-new database** that has never had Flyway run against it, `baseline-on-migrate: true` (set in `application.yml`) handles this automatically.  If you are restoring from a backup and migrations appear out of order, see the Flyway documentation on [repair](https://documentation.red-gate.com/fd/repair-184127628.html).
